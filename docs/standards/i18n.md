# Internationalisation (i18n)

Per ORCHESTRATOR §4 D5: **English only for v1**. Locale-loader and string-extraction wiring are scaffolded here; a second locale spike is owner-triggered.

## How strings work today

Strings live in `apps/web/src/lib/i18n/messages/en-US.json` as a flat key-to-value map. Keys are dot-namespaced by feature, e.g. `settings.heading`.

In a React component, import the helper:

```ts
import { t } from "@/lib/i18n";
```

Then use it inline:

```tsx
<h1>{t("settings.heading")}</h1>
```

If the key is missing, `t()` returns the key itself as a fallback and logs a one-time `console.warn` in dev so the missing string is visible without breaking the render.

## Adding a string

1. Add the key to `apps/web/src/lib/i18n/messages/en-US.json`.
2. Call `t("your.new.key")` at the use site.
3. Run `pnpm vitest run src/lib/i18n` to confirm.

## Adding a second locale (deferred)

When the owner triggers a second-locale spike:

1. Add `apps/web/src/lib/i18n/messages/<locale>.json` mirroring `en-US.json` keys.
2. Extend `MESSAGES` in `apps/web/src/lib/i18n/index.ts` to include the new locale.
3. Wire a locale-selector (cookie, header, or query string) into `t()` so it picks the right map.
4. Decide on a translation pipeline (manual JSON edits, weblate, etc.).
5. Revisit whether the no-dep helper is still enough or if `next-intl` / `react-i18next` is worth introducing for plurals + interpolation.

No locale switching, route restructuring, or extraction tooling ships in v1. The current shape keeps the surface tiny so the eventual swap is mechanical.
